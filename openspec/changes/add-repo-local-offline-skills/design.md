## Context

Проект использует набор личных workflow-skills из `~/.codex/skills`, но они не
входят в clone и поэтому не являются воспроизводимой зависимостью команды.
Принятый topology-контракт требует versioned process package вместо ручных
неотслеживаемых форков. Codex обнаруживает repo-local skills в
`.agents/skills/`.

## Goals / Non-Goals

**Goals:**

- хранить tool-agnostic источник в `process/agent-skills/`;
- предоставить Codex-проекцию в `.agents/skills/`;
- исключить глобальные skill dependencies и внешние операции;
- объяснить другим runtime создание отдельной проекции;
- сохранить human authority над внешними действиями.

**Non-Goals:**

- автоматические или поведенческие тесты skills;
- немедленное создание `.gigacode/skills/` и других непроверенных адаптеров;
- сетевой sandbox runtime;
- push, PR, публикация, deployment или внешняя интеграция.

## Decisions

1. `process/agent-skills/` является единственным каноническим источником.
   Прямая канонизация `.agents/skills/` проще, но выводит skill package из
   существующего `process/` distribution boundary.
2. `.agents/skills/` хранится в Git как точная проекция без symlink. Symlink
   исключён из-за Windows и архивной переносимости.
3. Имена получают префикс `teamssd-`, потому что Codex не объединяет skills с
   одинаковым `name`, а наличие глобальной и repo-local версии иначе создаёт
   неоднозначность.
4. Внешняя операция заменяется остановкой и описанием требуемого человеческого
   действия. Offline означает отсутствие внешних URL, web/MCP/connectors,
   remote Git, package installation и публикации.
5. По прямому решению владельца проверка ограничивается ручным статическим
   просмотром; тестовые файлы не создаются.

## Risks / Trade-offs

- [Канонический источник и проекция могут разойтись] → перед завершением
  сравнить хэши всех `SKILL.md`.
- [Текстовая offline-граница не заменяет runtime sandbox] → явно описать
  остаточный риск и не заявлять техническую сетевую изоляцию.
- [Без поведенческих тестов агент может иначе интерпретировать инструкции] →
  сохранить human review gate и зафиксировать решение владельца.
- [24 skills увеличивают объём сопровождения] → держать каждый skill
  самостоятельным и использовать один namespace.

## Migration Plan

1. Добавить OpenSpec delta и roadmap ownership.
2. Создать канонические `teamssd-*` skills и README.
3. Добавить Codex-проекцию.
4. Обновить package inventory и project guidance.
5. Выполнить ручные статические проверки и перевести change в
   `pending_acceptance`.

Rollback: удалить обе проекции, восстановить правило о глобальных skills и
откатить change одним Git revert; внешнее состояние не затрагивается.
