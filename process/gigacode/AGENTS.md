# Короткий контекст для GigaCode

Сначала примените `.gigacode/skills/superpowers.md`. Для SDD-задачи после него
примените `.gigacode/skills/sdd-process-companion.md`. Companion role gate и
authority boundaries имеют приоритет для конкретного SDD-действия.

Затем прочитайте корневой `AGENTS.md` и `PROCESS_MAP.md`. Они главнее этого
файла. Не загружайте весь репозиторий «на всякий случай».

## Главный чат

- До любого анализа требования, `intake`, вопроса о бизнес-ценности, создания артефакта или
  CTA установите роль человека. Если человек явно не назвал роль, это блокирующее состояние:
  не выводите статус процесса и не задавайте других вопросов; спросите только: «Какова ваша
  роль в этом чате: Analyst, Tech Lead, Developer, QA?».
  Не выводите роль из содержания требования и не считайте ответ интерфейса `Да` доказательством
  роли или человеческого acceptance. Интерактивные роли P3 — только Analyst, Tech Lead, Developer и QA: Analyst может дать доверенное acceptance спецификации, но не получает CTA на реализацию; ни одна интерактивная роль не получает CTA на реализацию: Analyst готовит PR для ролевого согласования, а Tech Lead только мониторит процесс.
- Держит только: роль человека, текущий этап, текущий change package и один
  следующий разрешённый шаг.
- Спрашивает человека, когда не хватает существенного факта или решения.
- Не вставляет в диалог длинные исходные документы: ссылается на путь и
  запрашивает короткую выжимку при необходимости.
- Использует `.gigacode/skills/superpowers.md` как общий workflow и
  `.gigacode/skills/sdd-process-companion.md` как SDD-сценарий работы.
- После существенного изменения пакета запускает
  `python scripts/guided_process_summary.py <change-package> --json` и не
  продолжает этап при `invalid`.
- В значимом ответе указывает пути к артефактам, ledger, проверки и ровно один
  следующий разрешённый шаг. Нельзя объявлять acceptance или менять этап без
  буквального сообщения человека.

## Отдельные чистые проверки

Если среда умеет запускать отдельные агенты или чаты, используйте их для
поиска, проверки документа и верификации. Давайте им узкую задачу и только
нужные пути. Главному чату возвращайте результат по форме из
`CONTEXT_REQUEST.md`, а не полный текст прочитанных файлов.

Если среда не умеет отдельных агентов, человек может открыть новый чат и
вставить тот же шаблон запроса. Это не меняет процесс и не даёт отдельному
чату право принимать решения.

Не создавайте реальные approvals, не меняйте статус пакета и не реализуйте
работу только по результату отдельной проверки.
